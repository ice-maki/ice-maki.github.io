pwn.college kernel security 学习

发表于2026-01-20|更新于2026-01-21|pwncollege
|总字数:4.5k|阅读时长:21分钟|浏览量:

题目来自pwn.college

linux内核版本:5.4.0 /x86_64

level1

内核模块漏洞函数：

device_write()函数:device_state[0]的值为strncmp（写入数据，password）+1

device_read()函数:device_state[0]的值为2则输出flag

attack:write写密码，read读取flag

level2

内核模块漏洞函数：

device_write：接收16字节数据与password对比，一样则printk输出flag

attack:write写密码，执行代码后dmesg查看flag

level3

内核模块漏洞函数:

device_write：接收16字节数据与password对比，一样则调用win函数

后门函数win:执行commmit_creds(prepare_kernel_cred(0))，进程权限变root

attack:write写密码，read读flag

level4

和level3.0一样，write改成iocti即可

attack:

level5

防护:无 KASLR，内核基址固定为 0xffffffff81000000

内核模块漏洞函数：

device_ioctl:函数在命令号为 1337 时，将用户传入的参数 arg 直接作为函数指针调用。允许任意内核函数执行

存在已知后门函数 win，可通过模块偏移或 /proc/kallsyms 获取其地址

attack:ioctl传入win函数地址，gcc编译执行即可。本来用的python，结果数太大了不支持

level6

防护：无kaslr
内核模块漏洞函数：

device_write：函数将用户输入的数据复制到内核缓冲区后直接作为代码执行，实现内核态的任意代码执行。

获取 prepare_kernel_cred 和 commit_creds 函数地址（通过特权模式查看 /proc/kallsyms）

attack：编写commmit_creds(prepare_kernel_cred(0))的shellcode通过write发送

level7

内核模块漏洞函数：

device_ioctl：当命令号为 1337 时，内核会从用户空间读取三个关键数据块：
1.Shellcode 长度（8字节）：指定后续 Shellcode 的实际大小
2.Shellcode 主体（最多 4096 字节）：将被复制到内核的全局缓冲区
3.执行地址（8字节）：一个函数指针，指向最终执行的代码位置

虽然用户可以直接指定执行地址，但为了执行自定义的提权代码，需要知道内核模块中 shellcode 全局缓冲区的地址：
1.bss 段基地址：0xffffffffc0002440（通过 IDA 静态分析获得）
2.shellcode 变量偏移：+0x8
3.shellcode 指针地址：0xffffffffc0002448
通过 gdb 调试（或在特权模式下）可以查看该指针存储的实际地址，例如：0xffffc90000085000。由于未启用 KASLR，此地址在系统运行期间保持不变。

level8

内核模块漏洞函数：

device_write：将用户传入的数据直接作为函数执行。可利用此漏洞执行 commit_creds(prepare_kernel_cred(0)) 提权
但模块在初始化时通过 proc_create(“pwncollege”, 384, 0, &fops) 创建了 proc 条目，无法直接访问

题目提供的 ELF 程序具备访问权限。该程序流程如下：
1:通过 read 接收输入数据并直接作为函数执行；
2:启用 seccomp 沙箱，仅允许 write 系统调用。

因此需要通过该进程向内核模块发送 payload，但由于沙箱限制无法直接读取 /flag，需先绕过 seccomp。
关闭 seccomp 的方法为修改当前进程的 task_struct->thread_info->flags：
1:thread_info->flags 位于 task_struct 起始处；(版本不一样也不一样，也可利用gdb查看相关结构体的偏移)
(gdb) p &(((struct task_struct*)0)->thread_info)
2:清除 TIF_SECCOMP 标志位：current->thread_info.flags &= ~(1 << TIF_SECCOMP)。
3:可通过 revert_creds, commit_creds 等使用 current 的函数，在 gdb 中定位其地址,通过 gs 寄存器获取current

attack：
1:构造内核 shellcode：关闭 seccomp 并提权；
2:构造用户态 shellcode：通过 write 将内核 shellcode 发送至模块，随后读取 flag。

level9

内核模块漏洞函数：

device_write:拷贝最多 0x108 字节数据到 0x100 字节缓冲区，导致栈溢出。log_function 指针位于缓冲区之后，可被覆盖。

结构体如下:

attack:
1.覆盖 log_function 为内核函数 run_cmd（类似 system）。
2.缓冲区前半部分填充待执行的 shell 命令。

level10

和level9是相同的内核模块，但是内核地址随机化（KASLR）启用，需先泄露地址计算偏移。

初始 log_function 指向 printk，调用时可泄露栈上数据。
对比无 KASLR 的已知地址（如 0xffffffff810b69a9）与泄露地址，差值即为偏移。
无KASLR情况泄露第38个参数地址为0xffffffff810b69a9
有KASLR情况泄露第38个参数地址为0xffffffffb9eb69a9
偏移=0xffffffffb9eb69a9-0xffffffff810b69a9

attack:利用px(lx)求出KASLR偏移，将log_function修改为指向run_cmd函数的地址

level11

内核模块漏洞函数：

device_write：传入的数据当成函数执行，访问权限384，无访问权限，需要利用给的elf访问漏洞模块

用户态elf：
1:通过 read 接收输入数据并直接作为函数执行；
2:print_disassembly函数是检查我们传入的数据是否是可执行函数
3:启用 seccomp 沙箱，仅允许 write 系统调用。

load_flag()它通过 fork() 创建了一个子进程，
由子进程将 /flag 文件的内容读取到其私有的内存地址 0x404040 中，随后主进程立即调用 unlink(“/flag”) 删除磁盘上的文件。

这意味着，flag 仅存在于子进程的虚拟地址空间内。由于进程隔离，主进程无法通过常规的 read 系统调用访问 /flag 文件，
也无法直接读取子进程的内存。同时，主进程执行的用户态 Shellcode 被严格限制，只能使用 write 系统调用。

内核模块的 device_write 函数提供了内核态的任意代码执行能力。虽然系统启用了 SMEP（防止内核执行用户态代码），
但 SMAP（防止内核访问用户态内存）并未开启。我们可以编写内核态的 Shellcode，
利用内核特权遍历子进程的页表，找到 flag 数据对应的物理内存地址，然后通过内核的直接映射区访问该物理内存，从而绕过进程隔离。

attack:
定位子进程的task_struct，遍历页表获取0x404040对应的物理地址，通过直接映射区读取flag并打印

可以先写一个获取物理地址的c代码然后编译，利用obj查看的汇编来，编写需要的汇编
c流程：
1.通过 current->pid + 1 推算出子进程的 PID。
2.调用内核函数 find_get_pid 和 get_pid_task 获取子进程的 task_struct 结构体。
3.从 task_struct 中找到 mm_struct (内存描述符)。
4.从 mm_struct 获取页全局目录 (PGD) 基地址。
5.对目标虚拟地址 0x404040 执行标准的页表遍历：PGD -> P4D -> PUD -> PMD -> PTE。
6.最终得到的 PTE（页表项）中提取出页帧号 (PFN)，结合页内偏移，计算出 0x404040 对应的完整物理地址。

反编译的到如下汇编,最后的rsi就是pte的值，提取pfn加上页内偏移，得到物理地址

1.Linux 内核将全部物理内存线性映射到虚拟地址空间 0xffff888000000000 起始的区域。
2.将计算出的物理地址加上该基址，即可得到内核可直接读写的虚拟地址。
3.使用 printk 内核函数，将读取到的 flag 内容输出到内核日志 (dmesg)。

所需函数地址如下:
find_get_pid:0xffffffff81084fd0
get_pid_task:0xffffffff81084de0
printk:0xffffffff810b69a9

手动添加获取直接映射区地址+printk输出

level12

唯一不同点在于load_flag函数：子进程读取完flag之后exit()结束了进程，映射关系会被删除
虽然页表被释放，但 flag 数据所在的物理内存页并未被清零。在内核中，为效率考虑，物理内存仅在分配给新页面时才会清零（惰性清零策略）

可以直接在直接映射区（0xffff888000000000 开始）中搜索包含 flag 特征字符串（pwn.college{）的内存区域。
由于 flag 刚被读取且进程已退出，其物理页很可能仍在直接映射区的低地址区域

方式一：使用 Python 启动挑战程序并发送攻击代码。但这样可能会因为 Python 运行时的内存操作而覆盖 flag。
方式二：将攻击代码保存为文件， 然后通过 cat 命令将文件内容作为输入传递给挑战程序， 这样挑战程序直接读取文件中的攻击代码，而不经过 Python 运行时。

为了最小化内存操作，避免 flag 所在的物理页被重用和覆盖：
1.使用纯汇编编写攻击载荷，避免 Python 解释器的运行时内存分配。
2.通过 cat 管道直接将预编译的 shellcode 传递给挑战程序。

attack:
内核shellcode:访问所有直接映射区，提取出所有pwn.college开头的字符串（实际实现中需要设置合理的扫描边界，避免无限循环或访问无效内存。）
用户态shellcode:调用write写入内核shellcode
执行 cat baby12.asm | /challenge/babykernel_level12.0 + dmesg | grep pwn.college{
